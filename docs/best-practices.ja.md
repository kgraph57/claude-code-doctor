# フロンティアモデル有効活用のベストプラクティス

Fable 5クラスのモデルとClaude Codeを組み合わせて大きな仕事をさせるとき、
このスキルの設計に焼き込んだ原則。環境監査は題材のひとつで、原則自体は
リサーチ・レビュー・大型改修に共通する。

「なぜ掃除スクリプトではなく健康診断なのか」という思想は
[`ai-checkup-philosophy.ja.md`](ai-checkup-philosophy.ja.md) に分けている。

## 1. 仕事の形を決めてから、モデルを撃つ

いきなり本体に「監査して」と言わない。先に決めるのは:

- **スコープ**（どこまで見るか・どこに触れないか）
- **出力の形**（findingsスキーマ・マトリクス・フェーズプラン）
- **権限の線**(read-onlyか・どこへの書き込みを許すか)

形が決まっていれば、あとは並列化・集計・可視化がすべて機械的に進む。
形が曖昧なまま撃つと、賢いモデルほど「それらしい长文」を返してきて構造化できない。

## 2. read-only並列ファンアウト＋上位モデルの統合

- 網羅は**安いモデルの並列**で: 10領域のスキャンは判断より作業。
  サブエージェントにはmodelを明示指定する（デフォルト継承だと全部最上位モデルに
  なってコストが跳ねる。これはこの監査の頻出指摘そのもの）
- 判断は**上位モデルの1本**で: マトリクス化・決定事項の抽出・矛盾の裏取りは
  メインループが担う。裏取りは「1コマンドで確認できる急所2〜3件」に絞る

## 3. 構造化出力を強制する

自由記述で受けると統合が人力になる。スキーマ（title / severity / effort /
detail / recommendation）で受ければ:

- 集計（件数・領域別分布）が機械的
- ダッシュボード生成がスクリプト1本
- 「事実」と「提案」が分離され、実行判断が下しやすい

## 4. 診断と治療を分離する（User Sovereignty）

- 診断フェーズは**完全read-only**。どれだけ確信があっても変更しない
- 決定事項は**OPTION A/B形式**で提示し、番号で答えられるようにする
- 「たぶん良い」「かもしれない」は承認ではない。曖昧なら止まって聞く
- 治療は1個ずつ: バックアップ → 実行 → その場で検証 → 1行報告
- 削除はしない。隔離フォルダ＋MANIFEST（元パス・理由・日付）で退避し、後日削除

権限ガード（permissionsやhook）に止められたら、迂回せずユーザーに報告する。
ガードが効いたこと自体が、環境が健全である証拠。

## 5. 成果物は「見せられる形」まで持っていく

- 長いレポートはMarkdownで終わらせず、**HTMLダッシュボード**にする
  （数字サマリ→決定事項→マトリクス→全指摘の折りたたみ）
- 生成したHTMLは**headlessブラウザで実レンダリングを検証**してから完成と言う
  （CSSバグは目で見ないと見つからない）
- 公開するなら**サニタイズ済みの共有カード**を別に作る。内部レポートには
  実パス・セキュリティの穴・コスト実額が入っており、そのまま外に出してはいけない

## 6. 常時ロードは家賃

- CLAUDE.md・rules・メモリ・MCPツールリストは、毎セッション先に支払うコスト
- スキルはprogressive disclosureで書く: 本文（SKILL.md）は手順の骨格だけ、
  詳細はreferences/に分離して必要時にReadさせる
- 監査で「常時ロード合計トークン」を必ず数値化するのはこのため。
  測らないと減らせない
